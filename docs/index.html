>_GO-RS REFERENCE

Go, compiled and run without a garbage collector. Source is lexed with Go's automatic semicolon insertion, parsed to an AST, lowered to fusevm bytecode, and executed on the same language-agnostic VM + three-tier Cranelift JIT that hosts zshrs, stryke, awkrs, elisp, and rubylang. No GC, no goroutine runtime, no go toolchain. In active development.

What it is

go-rs compiles Go source to fusevm instructions and lets the shared engine execute and JIT-compile them. It is a pure frontend: it lexes and parses Go — including automatic semicolon insertion — lowers the AST to a fusevm::Chunk, and runs it. There is no bespoke VM, no tree-walker, and no Go runtime.

Arithmetic and comparisons lower to native fusevm ops so the Cranelift JIT can block- and trace-compile hot loops. Go's numeric semantics are honoured statically: / truncates for int ÷ int (the compiler tracks each value's static numeric category and appends Op::TruncInt only there) and divides as floating point otherwise. String + (concatenation) and string ordering dispatch at runtime through fusevm's strict numeric hook.

It is another language hosted on fusevm, the shared bytecode VM and Cranelift JIT behind zshrs (the shell), stryke (the language), awkrs (AWK), elisp (Emacs Lisp), and rubylang (Ruby). go-rs carries no VM or JIT of its own.

Architecture

The pipeline mirrors how zshrs hosts zsh and rubylang hosts Ruby:

Go source  →  lexer (ASI)  →  parser (AST)  →  lower to fusevm bytecode  →  fusevm VM + Cranelift JIT
                                                   │
                              host ops: fmt print family + strict numeric hook (string + / ordering)

func main's body runs in the global scope (variables addressed by name via GetVar/SetVar). Every other func lowers to a subroutine, emitted after main and jumped over; its locals live in call-frame slots (GetSlot/SetSlot) so recursion never clobbers a caller's variables. Calls resolve by name index through Op::Call.

Why no GC

The reference Go toolchain ships a concurrent tri-color mark-sweep collector. go-rs targets the subset of Go whose values live on the fusevm value stack and in call-frame slots — integers, floats, booleans, and strings — so no tracing collector is needed for that surface. Heap-allocated aggregates (slices, maps, structs, interfaces) and their memory model are on the roadmap below; they will be reached through fusevm's extension-handle mechanism rather than a bolted-on GC.

Language surface

Status reflects the current v0.1.0 build. This is an early frontend: an unsupported construct is a parse or compile error, never a silent mis-run.

Build & run

git clone https://github.com/MenkeTechnologies/go-rs && cd go-rs
cargo build
./target/debug/go run hello.go
cargo test

fusevm is pulled from crates.io with the jit, jit-disk-cache, aot, and ffi features. The binary is named go; the library crate is gors.
