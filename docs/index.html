>_GO-RS REFERENCE

A Go frontend written in Rust. Source is lexed (with Go's automatic semicolon insertion) and parsed to an AST, lowered to fusevm bytecode, and executed on the same language-agnostic VM + three-tier Cranelift JIT that hosts zshrs, stryke, awkrs, elisprs, and rubylang. No bespoke VM, no goroutine runtime, no gc, no go toolchain. In active development.

What it is

go-rs runs Go programs as ordinary compiled bytecode: it lexes and parses the source, lowers it to fusevm instructions, and lets the shared engine execute and JIT-compile them. There is no bespoke VM or tree-walker — arithmetic and comparisons lower to native fusevm ops so the Cranelift JIT can trace hot loops, while Go-specific behaviour (the string + concatenation overload and lexicographic string ordering, Go's %v value formatting where a whole float64 prints as 3) is served by the host numeric hook and print builtins.

It is another language hosted on fusevm, the shared bytecode VM and Cranelift JIT behind zshrs (the shell), stryke (the language), awkrs (AWK), elisprs (Emacs Lisp), and rubylang (Ruby). go-rs carries no VM or JIT of its own — the go toolchain compiles Go to a native binary with a goroutine runtime and garbage collector; go-rs runs Go on fusevm.

Architecture

The pipeline mirrors how zshrs hosts zsh and rubylang hosts Ruby:

Go source  →  lexer (+ ASI)  →  parser (AST)  →  lower to fusevm bytecode  →  fusevm VM + Cranelift JIT
                                                    │
                                          host: string + concat, Go value formatting, fmt builtins

fusevm-hosted

No local vm.rs / jit.rs. Go is lowered to fusevm bytecode and executed on the shared three-tier Cranelift JIT; jit-disk-cache persists compiled native code across runs.

No go toolchain

go-rs never invokes go build and links no Go runtime. It is a source-in, run-out frontend: the compiled chunk runs directly on fusevm — no goroutine scheduler, no garbage collector.

Native fast paths

Integer and floating arithmetic and comparisons lower to native fusevm ops, so the tracing JIT can compile hot for loops; only string + and ordering take the host hook. Static typing makes int / int truncate.

Editor tooling

A read-only LSP (--lsp) supplies completion, hover, and parser-driven diagnostics; a DAP debugger (--dap) provides breakpoints, stepping, and local inspection over the running program.

Example

package main

import "fmt"

func fib(n int) int {
    if n < 2 {
        return n
    }
    return fib(n-1) + fib(n-2)
}

func main() {
    total := 0
    for i := 0; i < 10; i++ {
        total += fib(i)
    }
    fmt.Println(total)                 // => 88

    label := "sum = " + "done"
    fmt.Println(label)                 // => sum = done

    fmt.Println(6.0 / 2.0)             // => 3   (Go %v float formatting)
    fmt.Println(7 / 2)                 // => 3   (int division truncates)
}

Why go-rs

Compiled, not walked

go-rs lowers Go to fusevm bytecode and JITs hot paths through Cranelift — the same architecture that makes zshrs and stryke fast.

No go toolchain dependency

No go build, no module cache, no runtime install: a single go binary lexes, parses, lowers, and runs.

One shared engine

Bug fixes and JIT improvements in fusevm benefit zshrs, stryke, awkrs, elisprs, rubylang, and go-rs at once.

Editor-ready

LSP and DAP ship in the same binary, so completion, diagnostics, and a stepping debugger work from any editor that speaks the protocols.

Building from source

go-rs builds as a standalone Rust crate (it is not a workspace member of the meta repo):

# clone
git clone https://github.com/MenkeTechnologies/go-rs
cd go-rs

# build (produces target/debug/go)
cargo build

# run a program, or inspect the pipeline
./target/debug/go run hello.go
./target/debug/go --disasm fib.go
./target/debug/go --dump-ast fizzbuzz.go

# editor tooling over stdio
./target/debug/go --lsp
./target/debug/go --dap

# run tests
cargo test

fusevm is pulled from crates.io with the jit, jit-disk-cache, aot, and ffi features.
