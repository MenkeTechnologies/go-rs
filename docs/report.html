>_ENGINEERING REPORT

go-rs is a Go frontend for fusevm. It lexes and parses Go, lowers the AST to a fusevm::Chunk, and runs it on the shared bytecode VM and three-tier Cranelift JIT. It carries no VM, no JIT, and no Go runtime of its own — no GC, no goroutine runtime, no go toolchain. This report describes the architecture and the current scope. go-rs is in active development; the sections below track what is implemented today versus planned.

Position: a fusevm frontend

fusevm is a language-agnostic bytecode VM: any frontend compiles to the same Op enum and gets fused hot-loop dispatch, extension opcode tables, stack + slot execution, and an optional three-tier Cranelift JIT — for free. go-rs is one of those frontends, alongside zshrs, strykelang, awkrs, vimlrs, elisprs, rubylang, arb, pythonrs, phplang, node-js, rlang, and the JVM-language frontends. Every perf improvement in fusevm compounds across all of them, go-rs included.

Pure frontend, no runtime

go-rs owns the Go-specific front half — lexer, parser, compiler — and delegates all execution to fusevm. There is no tree-walking interpreter and no bespoke bytecode format. Arithmetic and comparisons lower to native fusevm ops, so the Cranelift JIT can block- and trace-compile hot loops without go-rs writing a single line of codegen.

Pipeline

Go source
   │  lexer.rs   — tokeniser with Go automatic semicolon insertion (ASI)
   ▼
tokens
   │  parser.rs  — recursive-descent parser → AST (ast.rs)
   ▼
AST
   │  compiler.rs — lower to fusevm ops via ChunkBuilder
   ▼
fusevm::Chunk ──► fusevm VM + three-tier Cranelift JIT
   │
   └─ host.rs   — fmt print family (Println/Print/Printf/Eprint*) + strict numeric
                  hook: string `+` (concat) and string ordering

func main's body runs in the global scope, with variables addressed by name through GetVar/SetVar. Every other func is lowered to a subroutine, emitted after main and jumped over; its locals live in call-frame slots (GetSlot/SetSlot) so recursion never clobbers a caller's variables. Calls resolve by name index through Op::Call.

Numeric semantics

Go's / truncates for integer operands and divides as floating point otherwise. The compiler tracks each value's static numeric category and appends Op::TruncInt only for int ÷ int; mixed and float operands divide directly. Integer and float arithmetic, comparisons, and boolean logic all lower to native fusevm ops. String + and string ordering are dispatched at runtime through the strict numeric hook in host.rs, so a string operand never silently coerces to a number.

Roadmap

go-rs currently covers a core imperative subset: package/import, func declarations and calls, var/const/:=, if/else, for/range, break/continue/return, integer/float/string arithmetic, and comparisons. On the roadmap, in rough order:

AreaPlan
switch / caseStatement-level switch lowering to jump tables.
Composite typesSlices, arrays, and maps as fusevm extension handles — reached without a tracing GC.
struct / interface / methodsUser-defined types and method dispatch over extension handles.
PointersAddress-of / dereference with an escape-free value model.
Concurrencygo/chan/select/defer — a design open question given the no-goroutine-runtime constraint.
Standard libraryA curated subset (fmt, strings, strconv, …) served natively.

Dependency posture

go-rs depends on fusevm (pulled from crates.io with the jit, jit-disk-cache, aot, and ffi features) and a small set of durable, widely-vetted crates. There is no dependency on the go toolchain, no cgo, and no goroutine runtime. The published crate excludes CI config, the integration-test corpus, and example programs; consumers get the source and the go binary.
