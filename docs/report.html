>_ENGINEERING REPORT

go-rs is a Go frontend in Rust, hosted on the fusevm bytecode VM and its three-tier Cranelift JIT. This report describes the architecture, the value model, the current state of the tree, and the dependency posture. The statements below are facts about the design and the manifest, not aspirational metrics.

Summary

The reference go toolchain compiles Go to a native binary with a goroutine scheduler and garbage collector. go-rs takes a different path: it lexes and parses Go to an AST — implementing Go's automatic semicolon insertion in the lexer — lowers that to fusevm bytecode, and runs it on a compiled VM with a Cranelift JIT. Arithmetic and comparison operators lower to native VM ops so the JIT can trace hot loops; Go-specific behaviour — the string + concatenation overload, lexicographic string ordering, and Go's %v value formatting — is served through the host numeric hook and a set of numbered print builtins. It joins fusevm alongside zshrs, stryke, awkrs, elisprs, and rubylang, and reuses that shared bytecode VM and JIT rather than shipping its own. There is no goroutine runtime and no garbage collector.

The current tree is a single-file package main slice: the lexer (with ASI) / parser, AST-to-bytecode lowering, := and var declarations, plain and compound assignment, three-clause / condition / infinite for with break/continue, if/else with an init clause, top-level func with recursion, post-increment/decrement, short-circuiting &&/||, string concatenation, and fmt.Println/Print/Printf. Static int/float typing makes int / int truncate while float / float stays exact. Composite types run on a host heap: slices and maps (literals, make, indexing, len/cap/append/delete, for … range) as reference types, and structs (literals, field access, methods) with Go value-copy semantics; interfaces dispatch dynamically on a value's runtime type; and the strings and strconv packages ship as builtins. On top of the runtime it ships editor tooling: a read-only LSP (--lsp) with completion, hover, and parser-driven diagnostics; a DAP debugger (--dap) with line breakpoints, stepping, a stack trace, and local-variable inspection; and inline rust { } FFI. Channels and goroutines are a future wave — they need a suspension/scheduling primitive fusevm does not yet expose, and are deliberately not shipped as a wrong single-threaded stopgap.

Hosting on fusevm

go-rs contains no virtual machine or JIT of its own. The execution path is:

Go source  →  lexer (+ ASI)  →  parser → AST  →  lower to fusevm bytecode  →  fusevm VM + Cranelift JIT
                                                    │
                                       host: string + concat, Go value formatting, fmt builtins

Shared engine

fusevm is pulled from crates.io with the jit, jit-disk-cache, aot, and ffi features. JIT and VM improvements land once and benefit zshrs, stryke, awkrs, elisprs, rubylang, and go-rs together.

Native arithmetic

Operators lower to native fusevm ops; a strict numeric hook supplies Go's string + concatenation and ordering only for the non-numeric operand cases the VM does not compute natively (src/host.rs).

Persistent native code

jit-disk-cache persists compiled machine code across runs, keyed by chunk hash, so hot Go loops trace-compile once and warm runs skip recompilation.

Editor tooling in-binary

The same go binary speaks LSP (--lsp) and DAP (--dap) over stdio, so completion, diagnostics, and a stepping debugger work from any editor that speaks the protocols.

Value model

The current slice runs directly on the fusevm value model — there is no go-rs object heap yet. Immediate values (int, float64, bool) ride through the VM as fusevm::Value; a string literal is a heap string value, and a rune literal is modeled as a one-character string. func main runs in the global scope, so its locals are addressed by name through GetVar/SetVar into the VM's global slot table — which is also what the --dap debugger reads to list a frame's locals. Every other func is lowered to a fusevm subroutine whose locals live in call-frame slots (GetSlot/SetSlot), so recursion never clobbers a caller's variables.

Two places need Go semantics the VM's default flavour does not supply. Printing lowers fmt.Println/Print/Printf to registered builtins that format through Go's %v rules — true/false, a whole float64 as 3 (not 3.0), +Inf/-Inf/NaN — rather than the shell-flavoured default. And Go's + is string concatenation when an operand is a string: once the strict numeric hook is installed, any operation with a non-numeric operand delegates to it, where + concatenates and </== order lexicographically, while non-+ arithmetic on a string is reported as a type error.

Compatibility & longevity

Reference semantics

Behaviour tracks go where the slice is implemented — string + concatenation, int / int truncation, a whole float64 printed as 3 — with expected outputs frozen in the integration tests, which run the built go binary and assert stdout.

Corpus-driven docs

The LSP completion/hover corpus is the single source of truth for docs/reference.html, so the static page and the language server never drift.

Cross-architecture

macOS aarch64 and Linux x86_64 / aarch64 via the Cranelift JIT; portable bytecode underneath.

Standalone crate

An explicit empty [workspace] keeps go-rs buildable on its own, independent of the meta repo.
